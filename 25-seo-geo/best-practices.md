# 16 — SEO & GEO — Best Practices

> The "why" behind the checklist, with sources. Patterns here are
> recommendations; the things that actually broke are in
> [`gotchas.md`](gotchas.md).

## 1. SSG/SSR over CSR for AI crawlers

AI-specific crawlers (GPTBot, ClaudeBot, PerplexityBot, OAI-SearchBot)
have simpler/weaker JS execution than Googlebot. Server-rendered HTML is
the safe default: the full content ships on first fetch, no hydration
required. An Astro SSG build scored a perfect technical/crawlability
result for exactly this reason. Reserve SSR/CSR for authed surfaces that
you exclude from indexing anyway.

- Source: Google "Understand the JavaScript SEO basics"
  <https://developers.google.com/search/docs/crawling-indexing/javascript/javascript-seo-basics>
- OpenAI GPTBot docs <https://platform.openai.com/docs/gptbot>

## 2. One `@graph`, stable `@id`s, generated not hand-authored

Schema.org's `@graph` lets one JSON-LD block describe many linked
entities. The winning pattern (shipped identically in three repos):

- A single `<script type="application/ld+json" id="graph">` per page.
- Canonical `@id`s for shared entities (`#organization`, `#website`,
  `#founder`) referenced from every page, so crawlers dedupe to one
  entity instead of seeing N competing copies.
- Generated by one builder fed from frontmatter — manual schema across
  100+ localized pages drifts within weeks.

Why it matters for GEO: knowledge-graph backends (Google, Perplexity)
recognize a unified entity across the site; fragmented inline blocks read
as ambiguous and get dropped.

- Schema.org `@graph`: <https://schema.org/docs/jsonldcontext.jsonld> and
  Google structured-data guidelines
  <https://developers.google.com/search/docs/appearance/structured-data/sd-policies>

## 3. The canonical / hreflang / sitemap triplet shares one URL builder

Most GSC "Google chose a different canonical" and "Alternate page with
proper canonical tag" errors come from three subsystems each computing a
URL their own way and disagreeing on a trailing slash. Generate all three
from **one** `route(key, lang, params)` function and set
`trailingSlash` once in config. Then canonical = hreflang self-ref =
sitemap loc, byte-for-byte.

- Google canonicalization
  <https://developers.google.com/search/docs/crawling-indexing/canonicalization>
- hreflang
  <https://developers.google.com/search/docs/specialty/international/localized-versions>

## 4. Multi-tenant: index the landing, `noindex` the tenants, in four layers

For multi-tenant SaaS the correct indexation surface is the marketing
host only. Tenant subdomains are near-identical (duplicate content) and
expose private data. Enforce `noindex` in depth because each layer does a
different job:

- `robots.txt Disallow: /` — controls crawl budget, not indexing.
- `X-Robots-Tag: noindex, nofollow` HTTP header — the signal Google
  trusts most; set it in middleware when a tenant context is detected.
- `<meta name="robots" content="noindex">` — HTML fallback.
- edge/routing — keeps auth pages off the public surface entirely.

Tenant discovery, if desired, is done via structured embeds
(LocalBusiness/Service) on an indexed marketplace page, not by indexing
tenant domains.

- Google `noindex` / robots meta
  <https://developers.google.com/search/docs/crawling-indexing/block-indexing>

## 5. GEO content is short, direct, structured, and self-contained

LLMs cite passages they can lift whole. The reliable shape:

- H1/H2/H3 phrased as the user's question.
- First sentence answers it directly; detail follows.
- `<dl>` definitions, comparison tables, before/after numbers — these are
  the most-cited shapes in AI Overviews and Perplexity.
- ~480–650 dense words per article; under 400 is too thin to be useful.

Pair it with **wedge positioning**: win specific long-tail queries
("menstrual migraine diary") rather than the generic head term
("migraine tracker") where apps and incumbents dominate. Answer engines
reward the narrow, deep page on a precise question.

- Background on GEO: Aggarwal et al., "GEO: Generative Engine
  Optimization" (KDD 2024) <https://arxiv.org/abs/2311.09735>

## 6. E-E-A-T: a real Person, a resolving sameAs, real reviews

- Author = a standalone `Person` node (`#founder`) with credentials and a
  `sameAs` that **resolves**. Article `author` points to the Person, not
  the Organization (a one-line change worth several E-E-A-T points).
- `Organization.sameAs` is the bridge to knowledge graphs (Wikidata,
  DBpedia). Empty = invisible. Broken = actively harmful.
- Ratings/reviews: only assert `aggregateRating` with visible, verifiable
  reviews. Otherwise prefer external platforms (G2, Trustpilot). A
  verified 4.7 elsewhere beats a claimed 4.9 on your own page.
- The last 30 GEO points are external: Reddit (heavily cited by
  assistants), YouTube (strongest Gemini/AIO signal — Google owns both),
  LinkedIn/GBP. These take months.

- Google's E-E-A-T / Quality Rater Guidelines
  <https://developers.google.com/search/docs/fundamentals/creating-helpful-content>
- Google review-snippet policy
  <https://developers.google.com/search/docs/appearance/structured-data/review-snippet>

## 7. llms.txt is a live, versioned product artifact

`/llms.txt` is a Markdown site summary for models: what you are, a short
context blockquote, your top 10–20 pages with one-line descriptions,
pricing, legal, contact. Keep numbers concrete. Critically, it **drifts**
— wire it to the same SSOT as the site (price, provider, feature counts)
and re-audit on every product change. Stale llms.txt feeds models wrong
facts about you.

- The proposal: <https://llmstxt.org/>

## 8. Validate, never assume

- Curl production with each bot UA — assume nothing about WAF/CSP
  behavior; a browser visit won't reveal a bot challenge.
- An automated JSON-LD scanner in CI (one `@graph`, no dangling refs,
  regression baseline) is the only thing that holds schema correct across
  100+ pages and 8 locales.
- Google Rich Results Test on each schema type — local mock validators
  miss what the real one flags.
- Verify security headers are actually *delivered* in production (curl),
  not merely declared in config.

- Rich Results Test <https://search.google.com/test/rich-results>
- IndexNow (fast Bing indexing) <https://www.indexnow.org/>

## 9. Internationalization is market research, not translation

Localized pages need local terminology, standard sizes, VAT, currency,
and locale-specific keyword research — not a literal translation. Each
locale gets a real static page, meta descriptions trimmed per language
(DE/IT/FR run longer than EN), and translation bundles pass a structural
parity check in CI (every key present, arrays same length).

- Google international targeting
  <https://developers.google.com/search/docs/specialty/international/managing-multi-regional-sites>
